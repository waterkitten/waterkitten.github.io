Designing a minting app

我们的 minting service tutorial 展示了如何使用 scaffold-eth 创建一个可以在以太坊上打造 nft 的简单 web 应用。

在本文章中，我们稍微换个角度来把事情讲清楚：为了帮助您更全面地了解打造 NFT 的问题，我们不讨论 nft 铸造的基本机制，而是来一起去思考成功打造一个铸币平台所要考虑的各个部分。如果你考虑从头打造一个 minting app，那这份高级架构概述能帮你完成这个决策过程。

# 前期做重点决策

在任何开发项目的早期设计阶段，您可以做的最重要的事情之一，就是确定您的设计中以后最固定最不容易更改的部分。对于区块链项目来说尤其如此，在区块链项目中，数据很多都是长期存在且不被轻易篡改的，而且发布的数据修改成本高昂。

# 使用什么平台？

在您开始构建您的 minting app 之前，您需要决定在哪个区块链平台上构建。这是一个复杂的决定，因为在你的设计思路去落地的时候，需要权衡技术、社会和业务方面的所有因素。当你决定一个平台时，这里有一些关键问题需要思考：

  • 该平台是否活跃且不断发展？这不仅仅考虑平台是否有人气那么简单；一个活跃的、不断发展的开发者社区对于帮助您解决构建过程中遇到的障碍至关重要。
  • 您相信该平台的长期前景吗？花时间和精力开发一个随着时间的推移不再相关的平台没有多大意义。
  • 新用户加入平台的成本和难度高吗？如果一个平台上面做开发成本低，但用户体验不太理想，那么后面将会很痛苦。
  • 该平台的技术上的限制对你想法的实现影响大吗，无论是现在还是长期？例如，NFT School 上的许多示例都使用以太坊，但交易限制和 gas fee 可能会影响你的开发。

# 你需要后端吗？

区块链网络上一个很棒的事情是，他们不需要一个中心化的点去协调和控制，完全可以由去运行一个去中心的应用。现今有些应用已经落地了，特别是在 defi 领域，应用程序运行所需的所有信息都可以存在于链上。

NFT 通常将链上数据与现实资产和链下的结构化元数据相结合，管理这些资产通常需要一定的服务支持（可以理解为后端支持）。例如，如果你为用户定制化铸造 nft 或者说你的 nft 在应用程序上被许多用户所预览，你可能需要建立一个用户的账号系统和一些一个 profile page（用户个人消息页面）去修改用户个人信息。你可能还需要一些能有反馈的元素，比如评论啊，赞赏之类的，这些消息在传统数据库中的写入成本很低，但对于上链来说就太昂贵了。

对于当今的大多数 NFT 应用程序，后端系统还是有必要的。但是，可能存在安全考虑；例如，直接与区块链交互时，通常最好使用 MetaMask 钱包，以便用户可以直接用私钥交易。

# 测试和基础设施

区块链的试错成本很高昂，可以在代码确保没有问题前必要不要在主网上尝试任何操作。可以多依赖测试来保证代码的运行符合预期。您可能需要针对不同标准来修改测试环境，例如速度或者环境的保真度。

测试时，可以采用集成测试和端对端的测试来补充单元测试，这个测试主要是基于你的 web app 和后端以及你的区块链真实平台环境之间的。理想情况下，应该可以在一个高仿真的环境中做完整个堆栈的测试。

# 合同设计

# 链上逻辑

除了 ERC-721 等标准提供的基本所有权和转移机制之外，大部分 NFT 不需要太多的自定义链上逻辑 。 但是，如果您正在设计一个更复杂的系统，您可能需要将一些自定义逻辑加入到您的合约中。

一般来说，在开发的过程中，只有当你想要修改区块链的状态且给一些些链上的状态进行输入时候，才可能要去编写链上的时间。例如，如果我们正在编写 CryptoKitties 的克隆，将用户的小猫集合读入 JavaScript 应用程序，写程序去判断猫咪是否能生 “小猫咪”，然后记录新交易以创建 “小猫咪 “，这是有风险的操作。这是因为区块链本质上是异步的，而且无法保证事物一致性。在这个例子中，当创建 “小猫咪” 的交易被处理时，它的主人可能已经卖掉了 “父母”。

通过在单个合约函数中捕获状态修改的读取和写入组件状态，您可以确保一切都在单个事务中执行，因此保证事务一致。这种方法的缺点是您可能还需要在链上存储更多状态，这在以太坊等通用链上的开发成本就会增加。对于复杂的合约机制来说，您可能需要研究其他项目一下，比如 Flow ，它最初是为了支持 CryptoKitties 用例而创建的，此后成为许多成功项目的基础，包括 NBA Top Shot 和其他备受瞩目的市场。

# 可升级性

大多数成功的 Web 应用程序都会不断更新修改，但智能合约在默认情况下是不可变的并且难以更改。在您部署到主网之前，请考虑使用类似于 OpenZeppelin Upgrades plugin 。 这增加了一个间接层，允许您将更新推送到已部署的合约，而无需所有用户连接到新的合约地址。尽管说会增加流程的复杂度以及更多的 gas 费，但这样会避免后续无法更新而造成的更大开销。

# 用户认证 / 授权体系

如果您的应用程序提供给多名用户使用，那您将需要一些方法来让他们登陆并跟踪状态。幸运的是，区块链帐户可以增加或替换传统的用户凭据，例如用户名和密码。

# 基于钱包的身份验证

可以将基于密码的身份验证替换为基于钱包的身份验证。这通过在应用程序的后端系统中记录用户的公共地址来实现。当用户想要登录时，后端会提供一个随机值（或 nonce）进行签名，然后用户使用 MetaMask 等钱包使用其私钥创建签名。后端验证签名是由正确的帐户创建的，然后更新随机数以防止其被重复使用。有关此过程的更多详细信息，请参阅以太坊上基于钱包的身份验证的社区指南

如果您不想自己编写整个登录流程，您还可以使用与您的区块链集成的外部服务。例如，Magic 基于开放的去中心化身份 (DID) 标准，通过电子邮件和其他方法提供无密码身份验证 。Magic 的文档 会教你如何在十几个区块链网络进行上述操作。

# 帐户和权限

并非所有用户帐户都是平等的，当您设计 minting app 时，您需要一种方法将管理操作限制为帐户子集。

在链上，这可以使用基于角色的访问控制来完成，你可以给特定角色分配具体的访问和控制权限。如果您使用的是以太坊或与 EVM 兼容的网络，OpenZeppelin AccessControl 合约是开始探索更多关于实施帐户和权限的好地方。

# 操作员帐户（operator）

除了为您的用户提供权限完善的账号外，您可能还希望代表您的用户执行某些操作 —— 例如，启动批量代币转移或其他智能合约操作。智能合约标准通常提供 operator 账户代表代币所有者处理代币的方法或者 API。比如 ERC-721 的 setApprovalForAll 函数 ，授予 operator 帐户转移该 owner 帐户拥有的任何代币的权限。

这是一项强大的功能，因此您需要格外小心特权帐户的凭据和权限管理。您还应该清楚地向您的用户讲清楚您的平台帐户将具有哪些功能以及如何使用它们。

# 结束

一旦你考虑了这些重要的设计决策，你就可以更好地开始构建和测试你的应用程序 —— 并将其发布给你的用户！
